# 治理框架 — 三省六部制

> 放射式多代理治理架构：祖 agent → 三省 → 六部

## 概述

三省六部制是 Hermes 的多代理治理架构。祖 agent（主 Agent）直接听令于用户，统筹三省；三省负责计划、审核、调度；六部作为执行单元直接对任务负责。**六部之间禁止横向通信，所有流转必须经过尚书省。**

## 常驻角色设计

三省六部是**常驻角色**，不随项目的开始和完成而创建或销毁。它们永久存在，任务来了就激活，任务完了就待命。

| 维度 | 按需创建 | 常驻角色（本设计） |
|------|---------|-----------------|
| 生命周期 | 任务来→创建→销毁 | 永久存在，任务分派 |
| 记忆 | 每次从零开始 | 积累跨任务经验 |
| 上下文 | 每次重新理解项目 | 持有持久上下文 |
| Token 成本 | 每次重载 harness 规则 | 规则已在 system prompt 里 |
| 熟练度 | 不变 | 越用越精准 |

技术上，六部不是临时 spawn 的子 AIAgent，而是**长期运行的 Agent 实例**——类似 GatewayRunner 缓存的 AIAgent，永久持有，按需激活。

## 外部工具权限管理

接入外部网页或软件（ComfyUI、Obsidian、浏览器等）时，权限分两层管理：

| 层 | 负责方 | 管什么 |
|---|--------|--------|
| **操作权** — 谁能调用这个工具 | 兵部 | 工具的连接、配置、调用权限 |
| **安全权** — 调用是否合规 | 刑部 | 风险评估、权限边界、异常拦截 |

**祖 agent 作为权限总管**，因为它直接听令于用户，是唯一知道"我信任什么工具"的角色。

```
用户 → 祖agent（授权接入哪些工具）
         ├── 兵部：拿到工具操作权（连接配置、调用权限）
         └── 刑部：拿到工具安全权（风险策略、越权检查）
```

三省和六部只在祖 agent 授权的范围内操作。

## 祖 agent — 总入口

祖 agent 是 Hermes 主 Agent，是用户与整个系统的唯一交互入口。

- **权限**: 完全开放记忆库，统筹三省，直接对三省下达指令
- **职责**: 接收用户指令 → 授权任务给三省 → 接收最终结果返回用户
- **不直接执行**: 不亲自写代码、不亲自操作工具，而是通过三省调度

## 三省核心职责

### 中书省 — 规划层

从问题库或祖 agent 授权的指令中形成具体任务：

1. **定义问题**: 从知识库中调用内置知识，基于问题形成假设
2. **收集信息**: 调用工具收集额外信息验证假设
3. **设计方案**: 基于知识库中的工作流，设计验证假设的动作 schema

**输出**: 方案 schema（含假设、验证动作、预期结果）

### 门下省 — 审核层

**A. 信息完整性检验** — 收集的信息是否完全

**B. 方案范围调整** — 通过调整 schema 放大或缩小范围：
- 检索网络上是否有相似案例或 GitHub 开源项目
- 按相似程度排列，给出不同方案的解决工作流
- 找到工作流之间的异同点

**C. 缺失度判断**:
- 缺失程度高 → 打回中书省（≤ 3 轮）
- 缺失范围在允许范围内 → 交付尚书省

### 尚书省 — 调度中枢

尚书省是整个执行层的**唯一中枢**，三重角色：

| 角色 | 说明 |
|------|------|
| **分发器** | 把大任务拆成小任务，按职责分给对应部门 |
| **裁判** | 判断六部交回的成果是否合格、是否返工 |
| **编排器** | 决定任务顺序、节奏、并行/串行关系 |

**输出**:
- **MD 文件** → 直接交付给用户（方案说明/结果汇报）
- **JSON 文件** → 给六部查看（任务合同/执行计划）
- **工作流文件** → 给六部执行（具体步骤）

**核心流程**: 接任务 → 拆任务 → 派任务 → 收结果 → 验结果 → 再分派 → 整合输出

## 六部核心职责

六部是六个直接对任务负责的执行子 agent，掌控着绝大多数网页或软件的操控。

| 部门 | 核心职责 | 关键词 | 具体动作 |
|------|---------|--------|---------|
| **工部** | 产出型执行 | "我要做出什么结果" | 写代码、改文件、跑脚本、调用插件、操作软件、生成内容、执行工作流 |
| **户部** | 信息资产管理 | "我有什么资源可用" | 资料收集、数据整理、知识库维护、文件归档、Obsidian 笔记整理、表格分析 |
| **礼部** | 表达与输出 | "怎么呈现结果" | 写总结、写报告、写说明、写邮件、写文档、整理成品、对外呈现 |
| **兵部** | 工具调用与自动化 | "怎么把动作跑起来" | 调用 ComfyUI、浏览器自动化、软件联动、工作流编排、批量任务执行、插件调度 |
| **刑部** | 安全合规校验 | "有没有风险和错误" | 权限检查、风险判断、错误检测、结果验证、测试、回滚、异常处理 |
| **吏部** | 调度治理与状态追踪 | "流程是否闭环" | 任务分派、角色管理、优先级排序、状态跟踪、记录、流程闭环 |

### 工部 vs 兵部 的区别

| | 工部 | 兵部 |
|--|------|------|
| 关注点 | **产出什么** (What) | **怎么跑起来** (How) |
| 例子 | 写一个 Python 脚本 | 把这个脚本接入 ComfyUI 工作流自动执行 |
| 产出 | 代码、文件、内容 | 执行结果、批量产出、自动化流程 |

### 尚书省 vs 吏部 的区别

| | 尚书省 | 吏部 |
|--|--------|------|
| 层级 | **总调度中心** | **运行管理中心** |
| 决策 | 大任务拆分、跨部门任务、全局流程 | 具体执行单、子任务、临时调整 |
| 例子 | 决定"先给户部收集素材，再给工部处理，最后给礼部出报告" | 跟踪"户部的素材收集任务已经超时，需要转派" |
| 权限 | 决定总体流程和资源分配 | 维护状态、重分配、优先级变更 |

## 六部与尚书省的放射式关系

### 为什么必须是放射式

| 问题 | 网状结构（六部互连） | 放射式结构（经过尚书省） |
|------|-------------------|---------------------|
| 责任 | 出错说不清谁负责 | 单链路，一眼看出 |
| 协作 | "你先改""你先等"扯皮 | 尚书省统一协调 |
| 流程 | 横向沟通导致任务变形 | 尚书省控制不变形 |
| 审计 | 网状沟通难以追踪 | 单链路可追溯 |

### 四种核心动作

```
尚书省 → 六部: 派发任务
六部 → 尚书省: 回传成果
尚书省 → 六部: 检查成果
尚书省 → 六部: 再次派发下一步
```

### 禁止规则

- 六部之间**不直接传任务**
- 六部之间**不直接交换意见**
- 六部之间**不直接要求对方改动**
- 所有跨部协调**由尚书省统一处理**

## Token 预算治理

| 阶段 | 负责方 | 内容 |
|------|--------|------|
| 预算制定 | 尚书省 | 本次任务允许花多少 token，是否需要节流，是否允许大模型重推理 |
| 实时计量 | 吏部 | 每次调用的 token 用量、耗时、模型类型、任务编号、责任人 |
| 风险控制 | 吏部 | 消耗过快/超阈值/反复重试 → 触发拦截和告警 → 上报尚书省 |

## 数据流

```
用户指令
  ↓
祖 agent: 理解意图，授权任务给三省
  ↓
中书省: 调知识库 → 形成假设 → 设计验证动作 schema
  ↓
门下省: 信息完整性检验 + 搜索相似案例 + 对比方案 + 缺失度判断
  ├── 缺失高 → 打回中书省 (≤3轮)
  └── 缺失可接受 → 交付尚书省
  ↓
尚书省: 整理输出 MD 给用户 + JSON/工作流给六部
  ↓
尚书省拆分任务 → 派发给对应六部
  ↓
六部执行 → 回传成果给尚书省
  ↓
尚书省验收 → 判断下一步/整合/返工
  ↓
尚书省: 整合最终结果 → 通过祖 agent 返回用户
```

## GovernanceDAG 状态机

```
PLAN → REVIEW → DISPATCH → EXECUTE(ministry_1) → EXECUTE(ministry_2) → ... → INTEGRATE → COMPLETE
  ↑        ↓
  └─REVISE─┘ (最多 3 轮)
```

所有状态转换写入 JSONL 审计追踪。只有合法的状态转换才被允许。

## 配置项

```yaml
# config.yaml - governance section
governance:
  department_model: null      # 各部门使用的模型
  ministry_model: null        # 六部使用的模型
  max_concurrent_ministries: 3
  max_revision_rounds: 3
  token_budget:
    enabled: true
    default_limit: 100000     # 默认 token 预算
    warning_threshold: 0.8    # 80% 时警告
    hard_limit: 1.2           # 120% 时强制拦截
```
