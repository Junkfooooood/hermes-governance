# 错误处理与容错

> 8 级错误分类 + 指数退避 + 凭证轮换 + 降级机制

## 概述

Hermes 的错误处理系统通过 8 级优先级分类管线识别 API 错误，然后根据错误类型选择恢复策略：重试、凭证轮换、上下文压缩或降级到备用提供商。

## 核心架构

```
API 错误
  ↓
classify_api_error() → ClassifiedError
  ↓
恢复策略决策
  ├── 重试 (jittered_backoff)
  ├── 凭证轮换 (CredentialPool)
  ├── 上下文压缩 (ContextCompressor)
  ├── 降级 (fallback_providers)
  └── Thinking signature 重试
```

## 实现原理

### 错误分类器 (`agent/error_classifier.py`)

**FailoverReason 枚举** (line 24):
`auth`, `auth_permanent`, `billing`, `rate_limit`, `overloaded`, `server_error`, `timeout`, `context_overflow`, `payload_too_large`, `model_not_found`, `provider_policy_blocked`, `format_error`, `thinking_signature`, `long_context_tier`, `unknown`

**ClassifiedError** (line 64):
```python
@dataclass
class ClassifiedError:
    reason: FailoverReason
    status_code: int
    provider: str
    model: str
    message: str
    retryable: bool
    should_compress: bool
    should_rotate_credential: bool
    should_fallback: bool
```

### 8 级分类管线 (line 313)

| 优先级 | 检查内容 |
|--------|---------|
| 1 | 提供商特定模式（Anthropic thinking signature, long-context tier） |
| 2 | HTTP 状态码 + 消息细化 (401/403/402/404/413/429/500/502/503/529) |
| 3 | 响应体错误码分类 |
| 4 | 消息模式匹配 (billing vs rate_limit vs context vs auth) |
| 5 | SSL/TLS 瞬态错误 → 重试为 timeout |
| 6 | 服务器断开 + 大 session → 上下文溢出 |
| 7 | 传输/超时启发 (ReadTimeout, ConnectionError) |
| 8 | 兜底: unknown (可重试+退避) |

### 重试策略 (`agent/retry_utils.py`)

`jittered_backoff(attempt, base_delay=5.0, max_delay=120.0, jitter_ratio=0.5)`:
- 去相关指数退避，防止惊群重试
- 单调计数器以 `time.time_ns()` 为种子

### 重试循环 (run_agent.py:9898)

```
while retry_count < max_retries:
  ├── 认证/计费错误 → CredentialPool 凭证轮换
  ├── 提供商特定 token 刷新 (Codex OAuth, Anthropic setup-tokens)
  ├── 上下文溢出/载荷过大 → ContextCompressor 压缩
  ├── 不可重试错误 → 激活 fallback_providers
  ├── Thinking signature 无效 → 剥离重试
  └── 429 限流 → 读取 Retry-After + 退避
```

### 上下文压缩器 (`agent/context_compressor.py`)

`ContextCompressor` 扩展 `ContextEngine` ABC:

1. **剪枝旧工具结果**（廉价，无 LLM 调用）— 替换为 1 行摘要
2. **保护头部消息**（前 N 轮）
3. **保护尾部消息**（~20K token 预算）
4. **摘要中间轮次**（结构化 LLM 提示词）
5. **后续压缩**迭代更新先前摘要

`ContextEngine` ABC: `update_from_response()`, `should_compress()`, `compress()`, `on_session_start()`, `on_session_end()`

第三方引擎可通过 `context.engine` 配置替换内置压缩器

## 数据流

```
API 调用返回错误
  ↓
classify_api_error(error)
  ↓
ClassifiedError {reason, retryable, should_*}
  ↓
┌─ retryable → jittered_backoff → 重试
├─ should_rotate → CredentialPool.rotate()
├─ should_compress → ContextCompressor.compress()
├─ should_fallback → fallback_providers 激活
└─ !retryable → 抛出异常
```
